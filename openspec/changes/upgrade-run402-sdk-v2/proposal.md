## Why

`@run402/sdk@2.0.0` ("Unified Apply") was published 2026-05-18 and is a breaking release; `2.0.1` followed within 10 minutes as a critical hotfix; `2.1.0` followed ~3 hours later, rewriting the asset namespace onto the same unified-apply substrate; `2.2.0` followed another ~6 hours after that, adding `assets.put()` to `@run402/functions` so edge functions get the helper from the in-function runtime instead of having to bundle the full SDK. The four releases together collapse every release write (database, functions, site, subdomains, routes, checks) AND every blob write into one engine accessed two ways: `r.project(id).apply(spec)` for declarative writes, and `r.project(id).assets.put / uploadDir / syncDir / prepareDir / putMany` (in deploy-time code) or `assets.put(key, source, opts)` from `@run402/functions` (in edge-function code) for ergonomic asset operations. Both route through `/apply/v1/plans` + `/content/v1/plans` underneath (deploy-time via `/apply/v1/plans → /apply/v1/plans/:id/commit`, in-function via `/apply/v1/service-asset-put`). The pre-2.0 wire endpoints we currently depend on are removed.

The production gateway is **already** on v1.48 — the [2.0.1 release notes](https://github.com/kychee-com/run402/releases/tag/v2.0.1) state that `cli-integration` testing against production caught the broken `/storage/v1/uploads/:id/complete` route returning 404. That means:

1. **Our deploy path is on borrowed time** — `scripts/_lib.ts:499` calls `r.deploy.apply()` (gone in 2.0). The next deploy against a 2.0 gateway fails on the missing surface.
2. **Our edge-function upload pipeline is already broken** — `functions/upload-asset.js`, `functions/upload-resource.js`, and `src/lib/storage-upload.ts` all raw-fetch `POST /storage/v1/uploads` and `POST /storage/v1/uploads/:id/complete`, which the v1.48 cutover removed. Member photo uploads, resource uploads, and admin asset uploads are returning 404 right now. (The demo reset cron likely masked the symptom because seeded photos are baked into the deploy bundle, not uploaded at runtime.)

Release-by-release: 2.0.0 shipped with `r.project(id).apply` itself silently issuing the retired completion call. 2.0.1 fixed that path but left `r.assets.put` / `initUploadSession` / `completeUploadSession` calling retired routes (silent breakage). 2.1.0 fixes `r.assets.put` (now routes through the apply substrate) and makes the legacy session methods throw `LocalError` with explicit migration hints (loud failure, no more silent breakage). The recommended hero today is `r.project(id).assets.put(key, source, opts)` for single uploads or `r.project(id).assets.uploadDir(path, opts)` for Node batches; both compose into the same plan/commit flow `r.project(id).apply({ assets: ... })` uses, with `assets.put` returning an `AssetRef` carrying the CDN URL directly.

`@run402/sdk@1.69.0` is currently pinned. Every call site that touches a removed surface must migrate in the same change because the SDK and the gateway are lockstep.

## What Changes

- **Pin `@run402/sdk` at `=2.2.0`** as a `devDependency`. Replaces the current `1.69.0` exact pin. 2.0.0 ships with the broken-completion bug; 2.0.1 fixes the apply hero but leaves the assets namespace silently broken; 2.1.0 fixes the SDK-side assets but leaves edge functions without an in-runtime helper (forcing them to bundle the full SDK); 2.2.0 ships `assets.put()` in `@run402/functions`, which is the missing piece for ergonomic edge-function uploads. CI and dev install via `npm install --before=null @run402/sdk@2.2.0` because the local `npm` cutoff filters out fresh packages (per `CLAUDE.md`).
- **Migrate `scripts/_lib.ts` from `r.deploy.apply(spec, opts)` to `r.project(id).apply(spec, opts)`**. The scoped client is the only public write surface in 2.0; the spec's `project` field is now optional (the scope binds it). `r.project(id)` is async (returns `Promise<ScopedRun402>`) so the call requires `await` at construction.
- **Migrate the edge-function upload pipeline off `/storage/v1/uploads`** to `assets.put(key, source, opts)` from `@run402/functions@2.2.0`. `functions/upload-asset.js` and `functions/upload-resource.js` currently raw-fetch the retired endpoint with `RUN402_SERVICE_KEY`. They drop the SHA-256 helper and three-step multipart dance, and call the in-function `assets.put` helper that the runtime injects (no SDK bundling, no custom credentials provider — the runtime handles service-key auth). `src/lib/storage-upload.ts` (browser-side) is rewritten to POST the file payload to `functions/v1/upload-asset` instead of hitting Run402 directly; one fetch instead of three, and the wire-shape logic stays server-side. Legacy `initUploadSession` / `getUploadSession` / `completeUploadSession` methods throw `LocalError` in 2.1.0+ with explicit migration hints.
- **Update upload tests** (`tests/unit/security-bug-25-uploads.test.ts`, `tests/unit/security-bug-28-storage-upload.test.ts`) so URL assertions match the new endpoint shape rather than the retired `/storage/v1/upload*` URLs.
- **Update the deploy spec** to require the 2.0.0 pin and the scoped-apply call shape. The existing requirement "Deploy uses Run402 v1.69 public paths" with its `r.deploy.apply` reference is rewritten against the new hero (`r.project(id).apply`).
- **Sweep stale docs** that still describe the pre-Astro-rewrite deploy flow (`node deploy.js` → `app.json` → `run402 deploy` CLI): `STRUCTURE.md` (lines 6, 14, 72), `CUSTOMIZING.md` (lines 114, 182, 198, 199), `docs/spec.md` (line 144), `openspec/specs/scheduled-functions/spec.md` (lines 37–63), `openspec/specs/deploy/spec.md` (line 71), and `scripts/README.md`'s "1.50.x" historical paragraph. Also refresh the user-memory file `project_demo_deploy.md` which still references `node deploy.js`.
- **Carry `@run402/functions` lockstep to `2.2.0`** in the edge-function bundles. 2.2.0 adds `assets.put()` to the in-function helper library — the missing piece that lets edge functions use the apply substrate without bundling the full SDK. Other helpers (`db` / `adminDb` / `getUser` / `email` / `ai`) are unchanged.
- **Not breaking** for end-users: site URLs, deploy contract, user-visible upload flow, and the cron schedules embedded in deployed functions are unchanged. This is an internal pin bump + call-site refactor that preserves all externally observable behaviors.
