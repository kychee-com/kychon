## Why

Kychon's central principle is that *an AI agent's API is SQL for config and file editing for code* ŌĆö but the engine breaks that contract for portal chrome. `custom_css` (and a few theme artifacts) are baked into first-paint HTML from a build-time static seed/snapshot and **never reconciled against the live `site_config` DB** ([chrome-bake.ts:286](../../../src/lib/chrome-bake.ts) ŌåÆ [Portal.astro:115](../../../src/layouts/Portal.astro)). An agent (or operator) that runs the SQL the API invites ŌĆö `UPDATE site_config SET value=ŌĆ” WHERE key='custom_css'` ŌĆö gets a **silent no-op**: the write persists, nothing reads it back, the page is unchanged short of a full re-port. A silent no-op is worse than an error for an agent: it can't tell "my write failed" from "my write isn't read," so it loops, hallucinates a cause, or works *around* the engine (e.g. recreating the masthead inside `custom_css` ŌĆö the exact wrong-direction repair [kychon-concierge#60](https://github.com/kychee-com/kychon-concierge/issues/60) had to ban). This was discovered during a live port (issue [#147](https://github.com/kychee-com/kychon/issues/147)).

The bake-then-rehydrate pattern that *would* fix this already exists in the engine ŌĆö theme tokens ([config.ts:507](../../../src/lib/config.ts), `applyTheme`) and `sections` ([page-render.ts:564](../../../src/lib/page-render.ts)) both bake first paint from the seed and then reconcile against live DB at runtime. `custom_css` simply never joined the club, and nothing structural prevents the next field from making the same mistake.

## What Changes

- **Apply `custom_css` at runtime from live `site_config`.** Give the injected chrome `<style>` a stable id and add an `applyCustomCss(siteConfig.custom_css)` step next to `applyTheme`/`applyBranding` in `config.ts`. The live `site_config` rows are *already fetched* on every load ŌĆö closing the loop is nearly free and requires **no rebuild**: edit DB ŌåÆ reload ŌåÆ change shows.
- **Make first paint match live config at build (no FOUC).** When a deploy runs against a live project (`KYCHON_ANON_KEY` / `KYCHON_PROJECT_ID` present), the chrome bake fetches `custom_css` (+ the other coherence-tracked fields) from the live config API and overrides the snapshot, falling back to the snapshot when the live fetch is unavailable ŌĆö preserving the zero-brand-flash first-paint guarantee.
- **Shrink the build-only ("redeploy") set to its irreducible core.** `custom_css` *and* the theme font families move to `runtime` (fonts: `applyTheme` ensures the Google-Fonts `<link>` at runtime when the family changes). Only `color_scheme` and `motion` ŌĆö which must be inlined before first paint to prevent a flash ŌĆö remain `redeploy`. Every remaining exception is a gotcha an agent must remember; the design minimizes them.
- **Introduce a config field-editability registry** ŌĆö a single typed source of truth (`src/lib/config-fields.ts`) declaring, per `site_config` key, whether it is `runtime` (reconciled live on next load) or `redeploy`, with a human-readable reason.
- **Add a root-cause guard test with a self-explaining failure** that fails CI when any `site_config` field is consumed only at build time without being declared `redeploy` ŌĆö naming the offending field, the consuming file, and the exact next action. The codebase teaches the next agent. This is the "doesn't happen again" mechanism.
- **Expose the registry as a co-located agent affordance** ŌĆö generate `/config-fields.json` at build (queryable next to `site_config` by an agent operating against the live site) plus a docs reflection ŌĆö both generated from the one typed registry, so an agent discovers a field's apply-mode *where it edits* instead of learning by failing.
- Treat the irreducible build-only fields (`color_scheme`, `motion`) as **declared `redeploy`**, not silent no-ops ŌĆö the engine surfaces "this needs a redeploy" rather than swallowing the edit.
- **Establish a consistency rule:** new live `site_config` fields follow the `applyTheme` shape (dedicated `#wl-ŌĆ”` element, same apply paths, registry declaration), so one learned pattern covers all of them.

## Capabilities

### New Capabilities
- `live-config-coherence`: The write-read coherence invariant for `site_config` ŌĆö the field-editability registry (each key declared `runtime` or `redeploy` with a reason), the principle that the build-time seed/snapshot is a *first-paint cache and never a parallel authority*, the CI guard that forbids undeclared baked-only fields, and the machine-discoverable affordance that lets an agent query a field's apply-mode.

### Modified Capabilities
- `config-driven-ui`: Adds a requirement that `src/lib/config.ts` applies `custom_css` at runtime from live `site_config` (mirroring the existing theme-injection requirement) so live `custom_css` edits publish on reload without a rebuild.

## Impact

- **Code**: `src/lib/config.ts` (new `applyCustomCss` + dedicated `<style id="wl-custom-css">` updater; `applyTheme` extended to ensure the font `<link>` at runtime), `src/layouts/Portal.astro` (stable id on the chrome custom-css `<style>`), `src/lib/chrome-bake.ts` + `src/seeds/index.ts` (live-config override at build when targeting a live project), `src/lib/theme/fonts.ts` (shared font-URL logic reused at runtime), new `src/lib/config-fields.ts` registry, generated `dist/config-fields.json`, new self-explaining guard test, `agent-docs` STRUCTURE.md / CUSTOMIZING.md reflection.
- **Behavior**: live `custom_css` and font-family edits take effect on next page load with no rebuild; cold first paint matches live config (no demo-brand flash/FOUT regression); the only fields that need a rebuild (`color_scheme`, `motion`) are declared and reported as such instead of silently ignored; an agent can `GET /config-fields.json` to learn any field's apply-mode before editing.
- **No DB schema change, no API/contract break.** Scope is Kychon-only ŌĆö the run402 edge-injection alternative (inject live config into the first byte at the edge) is explicitly out of scope here.
- **Tests**: live value overrides snapshot at build; graceful fallback to snapshot on fetch failure; runtime `applyCustomCss` updates the style element; `applyTheme` ensures one font `<link>` without duplication; guard test catches an undeclared baked-only field with a guided message; `/config-fields.json` matches the registry.
